Current API Structure
---------------------
Api/server.py
- App composition root.
- Equivalent to Program.cs.

Api/routes/query_routes.py
- Query controller endpoints.

Api/routes/system_routes.py
- System/status/source controller endpoints.

Api/models.py
- Request and response models.
- Equivalent to DTOs/view models.

Api/services/runtime_service.py
- API-facing service layer.
- Holds startup state, sessions, metadata cache, and query orchestration.

How The Layers Connect
----------------------
Controller-style routes
  ->
API service layer
  ->
Core modules
  ->
Ollama / index / metadata

Endpoint Map
------------
GET /health
- Simple alive check.
- Connected to: Api/routes/system_routes.py -> RuntimeService.health_payload()

GET /info
- Index statistics and active retrieval info.
- Connected to: Api/routes/system_routes.py -> RuntimeService.info_payload()

GET /config/runtime
- Current non-secret runtime settings.
- Connected to: Api/routes/system_routes.py -> RuntimeService.runtime_config_payload()

GET /sources
- List available source names from metadata cache.
- Connected to: Api/routes/system_routes.py -> RuntimeService.list_sources_payload()

GET /sources/{source_name}
- Show chunks from one source for archive inspection.
- Connected to: Api/routes/system_routes.py -> RuntimeService.source_chunks_payload()

GET /memory
- Read one session's memory.
- Connected to: Api/routes/system_routes.py -> RuntimeService.get_memory()

DELETE /memory
- Clear one session's memory.
- Connected to: Api/routes/system_routes.py -> RuntimeService.clear_memory()

POST /query/inspect
- Test retrieval before generation.
- Connected to: Api/routes/query_routes.py -> RuntimeService.inspect_query() -> Core.retriever + Core.prompt

POST /query
- Full query path for normal grounded answers.
- Connected to: Api/routes/query_routes.py -> RuntimeService.run_query() -> Core.agent -> Core.retriever -> Core.prompt -> Ollama

POST /query/stream
- Full query path with streaming tokens.
- Connected to: Api/routes/query_routes.py -> RuntimeService.stream_query() -> Core.agent -> Core.retriever -> Core.prompt -> Ollama

POST /query/explore
- Full query path using the object-explorer prompt.
- Connected to: Api/routes/query_routes.py -> RuntimeService.run_query(mode=explorer) -> Core.agent -> Core.retriever -> Core.prompt -> Ollama

Why This Split Exists
---------------------
ASP.NET Core
- Owns the product.
- Users, auth, sessions, settings, orchestration, storage, admin, mobile API.

FastAPI
- Owns the AI workflow.
- Retrieval, ranking, stitching, prompt building, grounded answer generation.

Ollama
- Owns inference only.
- It does not pick chunks.
- It answers from the prompt and chunks your Python code gives it.

Metadata Store Meaning
----------------------
The metadata store describes each chunk.

Examples:
- source/book name
- chapter
- file type
- chunk id
- text
- future tags like legion, faction, character, event

Why it matters:
- the vector DB says "these chunks are similar"
- metadata says "what they are"
- metadata powers filters, browsing, suggestions, and citations

Vector DB Meaning
-----------------
The vector DB stores embeddings so the system can do semantic retrieval.

Current:
- FAISS

Better future options:
- pgvector if one database story matters most
- Qdrant if dedicated vector service matters most
- Azure AI Search if enterprise cloud story matters most

Recommended path:
- keep FAISS now
- move later when app/product layer is stable

Personalization Goal
--------------------
When the user opens the UI, suggest useful things that really exist in the archive.

Examples:
- Ask about the Third Legion
- Best Chaos warband stories
- Continue your Siege of Terra thread

Easiest implementation path:
1. Precompute verified topic cards from metadata and chunks.
2. Tag chunks/sources by faction, legion, event, character, theme.
3. Track what each user opens, queries, saves, and reuses.
4. Rank suggestions by archive coverage + user interest + quality.

This should begin as metadata-driven and usage-driven, not model-driven.
